<objective>
Build the end-to-end magic-link auth slice that IS the Walking Skeleton's proven capability (AUTH-04 + AUTH-01, implementing D-01 passwordless magic-link + D-03 role-based routing): an unauthenticated `/` redirects to the admin sign-in; the admin submits their email → `signInWithOtp({shouldCreateUser:false})` (D-01); the emailed link hits `/auth/confirm` which `verifyOtp`s the `token_hash` into a session; root resolves the user to exactly one role via `auth.getUser()` → `app_users.role` and redirects by role to a placeholder `/admin` console (D-03); `proxy.ts` refreshes the session on every request.

Purpose: This is the thinnest real end-to-end path through the whole stack (routing → @supabase/ssr cookie auth → real Supabase OTP round-trip → real app_users read → role redirect). It satisfies SC-3 (role resolved server-side via getUser, exactly one role) and AUTH-04 (admin sign-in). After this plan, a real user can do something they could not before: sign in and reach their surface.
Output: Sign-in form + server action, confirm route handler, role resolver, role-based root redirect, placeholder admin console, proxy session refresh, and a Playwright sign-in smoke spec.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server action / route handler | Untrusted email input + magic-link token cross here |
| cookie/JWT → authz decision | Session must be revalidated server-side, not trusted raw |
| anonymous visitor → /admin | Role gate must block non-admins server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing / Elevation | JWT/cookie spoofing for authz | mitigate | `getCurrentRole()` uses `auth.getUser()` (revalidates JWT), never `getSession()`; grep-gate asserts no getSession in role.ts/proxy.ts (Tasks 1–2, SC-3) |
| T-03-02 | Elevation of Privilege | open signup via magic link | mitigate | `signInWithOtp({ shouldCreateUser:false })`; admin seeded by SQL (01-02 D-02); grep asserts the flag is false (Task 2) |
| T-03-03 | Elevation of Privilege | non-admin reaching /admin | mitigate | `app/admin/page.tsx` server-guard `getCurrentRole()` → redirect non-admins (Task 3) |
| T-03-04 | Spoofing | random logouts from logic between client-create and getClaims | mitigate | proxy.ts places no code between `createServerClient` and `getClaims()` (Supabase warning, Task 2) |
| T-03-05 | Input (V5) | malformed email to signInWithOtp | mitigate | server-side email-format validation in the action before the OTP call (Task 2) |
</threat_model>
