<objective>
The driver-invite vertical slice — the one genuinely new primitive in this phase. An admin enters a
driver's email + name + phone at `/admin/drivers`; the server action calls
`auth.admin.generateLink({type:'invite'})` (creates the auth user AND returns a set-password link
WITHOUT sending email), assigns `role='driver'` in `app_users`, stores the profile in
`driver_profiles`, and the console reveals the link inline for the admin to copy and hand off
(D-03 stub / D-04 manual delivery). The driver opens the link → `/auth/confirm?type=invite` →
`/set-password` → resolves to the driver role. No open signup exists.

Purpose: ONBD-05 (invite drivers), AUTH-03 (closed pool — invite is the only path to a driver account),
and NOTF-04 (invite "email" — STUBBED here as a copy-paste link; the Resend send wires in Phase 7).
Output: the `/admin/drivers` route (page + view + form + actions) reusing the existing
`/auth/confirm` → `/set-password` machinery (no new auth route), plus the Redirect-URLs allowlist
checkpoint (RESEARCH Pitfall 1) and the invite e2e.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin browser → RSC page / action | Role re-verified server-side; invite is an admin-only mutation. |
| server action → Supabase Auth (GoTrue admin API) | `generateLink` is a service-role admin call that creates an auth user. |
| generated invite link → driver browser | The set-password link is a credential-equivalent; the redirect base must be trusted + allowlisted. |
| driver → /auth/confirm → /set-password | Existing verified-OTP session boundary (Phase 1); `type=invite` already allowlisted. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-TMP7 | Tampering | open-redirect via attacker-controlled redirect base | mitigate | redirectTo uses the trusted `NEXT_PUBLIC_SITE_URL` constant, never the `Origin` header (WR-04); the URL must be in the Supabase Redirect URLs allowlist (Task 3 checkpoint). |
| T-02-EOP5 | Elevation of Privilege | self-promotion to driver / mass-assignment of role | mitigate | Role written explicitly server-side as the literal `'driver'`; never derived from client input. No open signup route exists — `generateLink({type:'invite'})` is the only account-creation path (AUTH-03). Invite action re-gated to admin. |
| T-02-ID5 | Information Disclosure | account enumeration on re-invite | mitigate | Generic admin-facing `driverAlreadyInvited` message; do not branch error copy on provider detail; mirror the forgot-password no-enumeration pattern. |

| T-02-TMP8 | Tampering | direct auth.users INSERT breaking sign-in | mitigate | Account created via the GoTrue admin API (`generateLink`), never a raw `auth.users` insert (0001 seed comment); preserves `auth.identities`. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installed this plan; all libs vetted in Phase 1 (RESEARCH Package Legitimacy Audit). slopcheck N/A. |
