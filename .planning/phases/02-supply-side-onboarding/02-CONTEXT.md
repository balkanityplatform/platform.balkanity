# Phase 2: Supply-Side Onboarding - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver admin-only, no-code CRUD for the supply hierarchy so a guest has a real
slug link to book against and a closed driver pool exists:

- **Companies → Properties → Destinations** CRUD (create/edit/list) entirely
  through the admin console — no code or DB edits (ONBD-01/02/03/06).
- **Destination** fields: slug, label, address, zone, airport, active, plus
  **price + commission** with a live "you keep" calculation (ONBD-03/04).
- **Driver invites** (AUTH-03/NOTF-04): admin invites a contractor driver who
  receives an invite, sets a password, and lands with the `driver` role. No open
  signup.
- **Admin-only RLS** on all supply tables; a unique URL-safe slug resolves to its
  destination.

**Out of scope this phase (belongs elsewhere):**
- Guest booking against the slug (`/pickup/<slug>`) — Phase 4.
- The transfer entity and any transfer-specific columns — Phase 4 (`wp_` module).
- Fully wired Resend email send + cap guardrails — Phase 7 (invite email is
  **stubbed** here, see D-03).
- Driver pool / my-run views — Phase 6.

**Carried forward from Phase 1 (locked — do not re-litigate):**
- Auth is **email + password** (AUTH-04). Reuse existing `/sign-in`,
  `/forgot-password`, `/set-password`, `/auth/confirm` flows + `getCurrentRole()`
  server-side gate (`auth.getUser()`, never `getSession()`).
- Seam (PLAT-01): platform-generic tables are **UNPREFIXED**; module tables use
  `wp_`. Per ROADMAP Phase 2 notes, companies/properties/destinations are
  **platform-generic** (keep transfer-specific columns out).
- Service-role writes via `createAdminClient()` (`server-only`); anon client on
  the browser.
- `active` boolean already specified on destinations.

</domain>

<decisions>
## Implementation Decisions

### Driver Invite Flow (AUTH-03 / NOTF-04)
- **D-01:** **Reuse the existing admin set-password flow.** Admin enters a driver's
  details → system creates the Supabase auth user + an `app_users` row with
  `role='driver'` → generates the invite-to-set-password link reusing the existing
  `/set-password` + `/auth/confirm` machinery. Driver sets password, signs in,
  lands on `/driver`. Same code path as admin onboarding — no new auth pattern.
- **D-02:** Capture **email + name + phone** at invite time. (Storage location —
  on `app_users` vs a driver-profile table — is the planner's call; see Claude's
  Discretion. Keep it platform-generic, not `wp_`.)
- **D-03:** The invite **email is STUBBED** in Phase 2 (logged/generated, not
  sent). The full Resend send + cap guardrails land in Phase 7 (NOTF-04).
- **D-04:** Delivery during the pilot = **show the generated set-password link in
  the admin console** for the admin to copy and send manually. No email
  dependency to onboard the 3 pilot drivers.

### Pricing & Commission (ONBD-04)
- **D-05:** Commission is a **percentage of the destination price** (e.g. 15%),
  representing what the referring company earns (recorded in v1, paid out later
  via Stripe Connect — deferred). Rounding rules are the planner's call.
- **D-06:** The live calculation **shows both lines**: the commission amount, then
  **"you keep (before fees)" = price − commission** (Balkanity's net), plus a
  small note with the **estimated Stripe fee** (1.5% + €0.25 EEA per CLAUDE.md
  Verified Provider Facts). The fee line is an estimate/note only — no fee logic
  is wired here (that's Phase 3).
- **D-07:** Price + commission are entered and displayed in **EUR**. (Stripe
  settlement currency is still formally confirmed in Phase 3, but EUR is the
  working display currency — matches the EEA fee facts and BG euro adoption.)
  Store money as integer minor units (cents).

### Slug Behavior (ONBD-03)
- **D-08:** Slug is **auto-generated by slugifying the label** on create, shown in
  an **editable field** so the admin can override it. Validate URL-safe + unique
  on save.
- **D-09:** Slug is **globally unique** across the platform (DB unique
  constraint) — `/pickup/<slug>` maps to exactly one destination, no property
  context in the URL.
- **D-10:** Slug is **editable after creation, with a UI warning** that changing
  it breaks any `/pickup` links already shared with guests (old slug 404s).

### Delete & Lifecycle
- **D-11:** **Soft-deactivate is the default** removal across all three levels via
  the `active` flag (preserves history + referential integrity for transfers that
  reference a destination). An inactive destination stops resolving its `/pickup`
  link. A **true hard delete is allowed only when the entity has no children and
  no transfers** referencing it.
- **D-12:** Deactivating a parent is **blocked while it has active children** —
  the admin deactivates bottom-up (destinations → property → company). This keeps
  the rule explicit and prevents live links lingering under a "removed" company.

### Claude's Discretion (planner/UI decide)
- Console layout / information architecture for navigating the
  company→property→destination hierarchy (drill-down vs flat-with-filters).
- Where driver name/phone are stored (extend `app_users` vs a separate
  platform-generic profile table) — keep platform-generic, no `wp_` prefix.
- Percentage rounding/precision rules and input validation specifics.
- Exact admin-only RLS policy shape (reuse the Phase 1 pattern: RLS enabled,
  reads/writes gated to admins; writes via service-role where appropriate).
- Slugify implementation + collision-suffix strategy on auto-generate.

</decisions>

  server-only; used for the invite's auth-user creation + role assignment.
- **`getCurrentRole()`** — admin gate for every console page/route (mirror
  `app/admin/page.tsx`'s `role !== "admin" → redirect("/sign-in")`).

  add console + form copy keys here (PLAT-04).
- **Design seed** — `Button` (52px CTA), `StatusDot` (active/inactive indicator),
  `LanguageToggle`. Inputs/cards/tables are NOT yet built — this phase is the
  first to need form + list UI (build per-phase, per Phase 1 D-08).

### Established Patterns
- Server Component page + client form "island" (e.g. `SignInForm`) so dictionary
  copy resolves server-side with no flash — mirror for create/edit forms.
- Server Actions in `actions.ts` colocated with the route for mutations.
- Module DB tables use `wp_` prefix; **platform-generic tables (companies,
  properties, destinations, driver profile) are UNPREFIXED** (PLAT-01 seam).

### Integration Points
- New admin console routes under `app/admin/*` (companies/properties/destinations
  CRUD + driver invites), all behind the `getCurrentRole()` admin gate.
- New migration(s) under `supabase/migrations/` (FLAGGED — sign-off before apply;
  Balkanity ref only).
- Slug resolution surface that Phase 4's `/pickup/<slug>` will consume (this phase
  creates the slug + uniqueness guarantee; Phase 4 reads it).

<specifics>
## Specific Ideas

- The whole point of the phase: a **second company can be onboarded end-to-end
  through the UI with zero code/DB edits** (ONBD-06) — the acceptance bar for
  "no-code".
- Slugs are guest-facing, shareable `/pickup/<slug>` URLs — keep them clean
  (label-derived, admin-tunable), which is why D-08/D-10 favor editability.
- Commission is **recorded, not paid** in v1 (Stripe Connect payout deferred) —
  the "you keep" UI is informational/record-keeping, not a settlement engine.

</specifics>
