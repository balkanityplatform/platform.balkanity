# Phase 2: Supply-Side Onboarding - Research

**Researched:** 2026-06-18
**Domain:** Admin-only no-code CRUD (companies → properties → destinations), driver-invite via Supabase Auth admin API, slug uniqueness, percentage commission math, admin-only RLS, schema migration push
**Confidence:** HIGH (driver invite, RLS, schema, push workflow verified against installed source + CLI; slug/money are reasoned design with HIGH confidence)

## Summary

This phase is the first to build real CRUD UI and a second schema migration. The genuinely new/unfamiliar bits are: (1) generating a driver set-password link **server-side without sending an email**, (2) globally-unique URL-safe slugs, and (3) percentage-commission math with a live "you keep" panel. Everything else reuses Phase 1 primitives verbatim (the three-client split, `getCurrentRole()` admin gate, the `/auth/confirm` → `/set-password` machinery, the EN/BG dictionary, the migration template).

The single most important verified finding: **`supabase.auth.admin.generateLink({ type: 'invite', email, options: { redirectTo } })` both creates the auth user AND returns `properties.action_link` without sending any email** — this is the exact primitive D-01/D-03/D-04 require (stubbed email, admin copies the link). It is confirmed in the installed `@supabase/auth-js` type definitions (the `generateLink()` JSDoc: *"handles the creation of the user for signup, invite and magiclink"*) and the official docs. The `redirectTo` must point at the existing `/auth/confirm?type=invite` route, and that URL must be added to the Supabase project's **Redirect URLs allowlist** or the link silently falls back to Site URL.

The schema is straightforward platform-generic (UNPREFIXED) FK hierarchy. The RLS pattern mirrors `0001` exactly: RLS enabled, admin-gated read policy, **no anon/authenticated write policy** — all writes go through `createAdminClient()` (service-role bypasses RLS) inside server actions that are themselves gated by `getCurrentRole()`. Money is integer minor units (cents), commission is an integer basis-points-or-percent column; the "you keep" calc is pure display arithmetic with banker's-safe integer rounding.

**Primary recommendation:** Reuse `generateLink({type:'invite'})` for the driver pool, mirror the `0001` admin-only-RLS + service-role-write pattern for three UNPREFIXED tables, hand-roll a ~15-line `slugify()` (no new dep) with a DB unique constraint + collision-suffix retry, and store all money as integer cents with display-only commission/fee math. Add one `[BLOCKING]` `supabase db push --linked` task (Balkanity ref ONLY).

**Pricing & Commission (ONBD-04)**
- **D-05:** Commission is a **percentage of the destination price** (e.g. 15%), representing what the referring company earns (recorded in v1, paid out later via Stripe Connect — deferred). Rounding rules are the planner's call.
- **D-06:** The live calculation **shows both lines**: the commission amount, then **"you keep (before fees)" = price − commission** (Balkanity's net), plus a small note with the **estimated Stripe fee** (1.5% + €0.25 EEA per CLAUDE.md Verified Provider Facts). - **D-07:** Price + commission entered and displayed in **EUR**. Store money as integer minor units (cents).

**Slug Behavior (ONBD-03)**
- **D-08:** Slug **auto-generated by slugifying the label** on create, shown in an **editable field** so the admin can override. Validate URL-safe + unique on save.
- **D-09:** Slug is **globally unique** across the platform (DB unique constraint) — `/pickup/<slug>` maps to exactly one destination, no property context in the URL.
- **D-10:** Slug is **editable after creation, with a UI warning** that changing it breaks any `/pickup` links already shared (old slug 404s).

**Delete & Lifecycle**
- **D-11:** **Soft-deactivate is the default** removal across all three levels via the `active` flag. An inactive destination stops resolving its `/pickup` link. A **true hard delete is allowed only when the entity has no children and no transfers** referencing it.
- **D-12:** Deactivating a parent is **blocked while it has active children** — the admin deactivates bottom-up (destinations → property → company).

| Slug uniqueness enforcement | Database (unique constraint) | API (slugify + collision retry) | The DB constraint is the authority; the app pre-slugifies + suffixes to avoid round-trips |

### Pattern 1: Driver invite (server action, service-role) — THE new primitive
**What:** Create the driver auth user + role row + profile, and return a copy-paste set-password link, **without sending email**.
**When to use:** The "Invite driver" action (ONBD-05 / AUTH-03 / NOTF-04-stub).
**Example:**
```typescript
// app/admin/drivers/actions.ts  "use server"
// Source: @supabase/auth-js GoTrueAdminApi.generateLink (installed types) +
//         supabase.com/docs/reference/javascript/auth-admin-generatelink
import { getCurrentRole } from "@/platform/auth/role";
import { createAdminClient } from "@/platform/supabase/admin";

export async function inviteDriver(_prev, formData: FormData) {
  if ((await getCurrentRole()) !== "admin") return { status: "error" /* generic */ };

  const parsed = inviteSchema.safeParse({
    email: formData.get("email"),
    name: formData.get("name"),
    phone: formData.get("phone"),
  });
  if (!parsed.success) return { status: "error", message: t.fieldRequired };

  const admin = createAdminClient();
  const base = process.env.NEXT_PUBLIC_SITE_URL!; // trusted base, NOT Origin header (WR-04)

  // generateLink(type:'invite') CREATES the auth user AND returns the link — sends NO email.
  const { data, error } = await admin.auth.admin.generateLink({
    type: "invite",
    email: parsed.data.email,
    options: {
      // must be in the project's Redirect URLs allowlist (see Pitfall 1)
      redirectTo: `${base}/auth/confirm?type=invite`,
      data: { name: parsed.data.name }, // optional user_metadata
    },
  });
  if (error || !data?.user) return { status: "error", message: t.saveFailed };

  // Assign the driver role + store profile (platform-generic, service-role write).
  await admin.from("app_users").insert({
    id: data.user.id, email: parsed.data.email, role: "driver",
  }); // (or upsert; handle re-invite of an existing email — see Pitfall 4)
  await admin.from("driver_profiles").insert({
    user_id: data.user.id, name: parsed.data.name, phone: parsed.data.phone,
  });

  // D-04: surface the link for manual copy; D-03: no email sent in Phase 2.
  return { status: "ok", actionLink: data.properties.action_link };
}
```
> Note: `data.properties` contains `action_link`, `hashed_token`, `email_otp`, `verification_type`, `redirect_to` [VERIFIED: installed `GenerateLinkProperties` type]. The existing `/auth/confirm` route already allowlists `type=invite` and forwards to `/set-password` — **no new auth route needed.**

### Pattern 2: Admin CRUD = RSC page (anon read) + client island + server action (service-role write)
**What:** Mirror the Phase 1 SignInForm shape: server component resolves dictionary copy with no flash, renders a client form island, mutations live in colocated `actions.ts`.
**When to use:** Every company/property/destination create/edit/list surface.
**Example:**
```typescript
// app/admin/companies/page.tsx  (RSC)
import { redirect } from "next/navigation";
import { getCurrentRole } from "@/platform/auth/role";
import { createClient } from "@/platform/supabase/server"; // anon, cookie-bound, RLS-scoped

export default async function CompaniesPage() {
  if ((await getCurrentRole()) !== "admin") redirect("/sign-in");
  const supabase = await createClient();
  const { data: companies } = await supabase
    .from("companies").select("id,name,active").order("name");
  return <CompaniesView companies={companies ?? []} /* + dictionary copy */ />;
}
```
> Reads use the **anon cookie-bound client** so the admin-read RLS policy is exercised (defence-in-depth). Writes use `createAdminClient()` inside the server action (no anon write policy exists by design).

### Pattern 3: Slug generation + DB-unique + collision suffix
**What:** Auto-slugify on label input (client), validate URL-safe + unique on save (server + DB).
**Example:**
```typescript
// platform/slug/slugify.ts
export function slugify(input: string): string {
  return input
    .normalize("NFKD").replace(/[̀-ͯ]/g, "") // strip Latin diacritics
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")  // non-url-safe → hyphen
    .replace(/^-+|-+$/g, "")      // trim leading/trailing hyphens
    .replace(/-{2,}/g, "-")       // collapse runs
    .slice(0, 80);
}
// Server action: ensure global uniqueness with a suffix retry.
async function uniqueSlug(admin, base: string): Promise<string> {
  let candidate = base || `dest`; let n = 1;
  // loop: SELECT 1 FROM destinations WHERE slug = candidate
  while (await slugTaken(admin, candidate)) candidate = `${base}-${++n}`;
  return candidate;
}
```
> The DB **unique constraint is the authority** (race-safe). The app pre-checks to give a clean inline error; on a unique-violation (`23505`) the server action must catch and re-suffix or return `t.slugTaken`. [VERIFIED: pattern; HIGH]

### Pattern 4: Money & commission (integer cents, display-only)
**What:** Store `price_cents integer`, `commission_pct` (integer percent or basis points). Derive commission/net at render.
**Example:**
```typescript
// platform/money/commission.ts  — cents in, cents out; round half-up.
export function commissionCents(priceCents: number, pct: number): number {
  return Math.round((priceCents * pct) / 100);       // pct = whole percent (e.g. 15)
}
export function netCents(priceCents: number, pct: number): number {
  return priceCents - commissionCents(priceCents, pct);
}
// Display-only estimate note (D-06) — EEA 1.5% + €0.25, CLAUDE.md Verified Provider Facts.
export function estStripeFeeCents(priceCents: number): number {
  return Math.round(priceCents * 0.015) + 25;
}
export const fmtEur = (cents: number) => (cents / 100).toFixed(2);
```
> Recompute live in the browser island on price/pct change. **Do not persist** commission/net/fee — they are derived. The fee line is an estimate label only; no fee is applied here (Phase 3). [VERIFIED: CLAUDE.md Verified Provider Facts — EEA 1.5% + €0.25]

### Anti-Patterns to Avoid
- **Anon write policy on supply tables:** Do NOT add INSERT/UPDATE policies for `authenticated`. Mirror `0001` — writes only via service-role inside admin-gated actions. A stray write policy widens the attack surface.
- **`inviteUserByEmail()` for the driver invite:** sends a real email (wrong for D-03 stub; wrong template on free tier). Use `generateLink({type:'invite'})`.
- **Trusting the `Origin` header for `redirectTo`:** use `NEXT_PUBLIC_SITE_URL` (the codebase already does this in `forgot-password/actions.ts`, WR-04) — header-controlled redirect is an open-redirect vector.
- **UI-only deactivation gate (D-12):** enforce "no active children" in the server action (clear error) AND back it with FK `on delete restrict`; never rely on hiding a button.
- **Floating-point euros:** never store/compute money as float. Integer cents only.
- **`getSession()` for the admin gate:** always `getCurrentRole()` → `auth.getUser()` (CLAUDE.md "What NOT to Use").
- **Prefixing supply tables with `wp_`:** companies/properties/destinations/driver_profiles are platform-generic — UNPREFIXED (PLAT-01).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Invite token / set-password link | Custom token table + email link verifier | `auth.admin.generateLink({type:'invite'})` + existing `/auth/confirm` | GoTrue issues a verifiable hashed OTP, sets `invited_at`, integrates with the session cookie machinery you already have |
| Driver auth account creation | Direct `auth.users` INSERT | `generateLink`/`createUser` admin API | A raw insert skips `auth.identities` and breaks sign-in (this exact gotcha is documented in `0001`'s seed comment) |
| Slug uniqueness | App-side "check then insert" only | DB **unique constraint** (authority) + app pre-check | Race-safe; two concurrent creates can't both win |
| Money precision | Float euros, client-trusted totals | Integer cents + server-side recompute | Stripe (Phase 3) consumes minor units; no rounding drift |
| Password rules / session | New password form/flow | Existing `/set-password` + `setPassword()` action | Already built, JWT-revalidating, dictionary-keyed |

**Key insight:** The driver invite feels like a new feature but is 90% wiring of primitives that already exist — the *only* new call is `generateLink`. Everything downstream (`/auth/confirm` → `/set-password` → role-route) is built and already allowlists `type=invite`.

## Common Pitfalls

### Pitfall 1: `redirectTo` not in the allowed Redirect URLs → invite lands on the wrong page
**What goes wrong:** `generateLink({options:{redirectTo}})` requires the URL to match the project's Redirect URLs allowlist (supports wildcards). If unconfigured, Supabase ignores `redirectTo` and uses Site URL, so the driver verifies but doesn't reach `/set-password` cleanly.
**Why it happens:** The allowlist is project config (dashboard), not code, and is easy to forget.
**How to avoid:** Add `${SITE_URL}/auth/confirm`, `${SITE_URL}/auth/confirm?type=invite`, and the localhost dev equivalents to **Authentication → URL Configuration → Redirect URLs** in the Balkanity project. Include a `checkpoint:human-verify` task for this. [VERIFIED: official docs + supabase/auth#1738]

### Pitfall 2: Cyrillic / non-Latin labels slugify to empty
**What goes wrong:** `normalize('NFKD')` strips Latin diacritics but does NOT transliterate Cyrillic; a fully-Cyrillic label (e.g. "Слънчев бряг") collapses to `""`, then to `"dest"`, then collides.
**Why it happens:** Hand-rolled slugify has no Cyrillic→Latin map.
**How to avoid:** When `slugify(label)` returns empty, fall back to a generated base (`dest` + collision suffix, or a short id), and rely on the editable slug field (D-08) so the admin can type a clean Latin slug. Document this; the pilot destinations (Sunny Beach, etc.) are typically typed in Latin anyway. [ASSUMED: pilot labels are Latin/English — confirm with user]

### Pitfall 3: Slug edited after sharing breaks `/pickup` links (D-10)
**What goes wrong:** Changing a live slug 404s every previously-shared guest link.
**How to avoid:** This is accepted behaviour (D-10) — the mitigation is the **coral UI warning** (UI-SPEC copy `Changing this link will break any /pickup links…`). No redirect/alias table in v1 (deferred). Warning must be shown before save in the slug edit path.

### Pitfall 4: Re-inviting an existing email
**What goes wrong:** `generateLink({type:'invite'})` on an email that already has an auth user errors (user exists), and a blind `app_users` insert hits the `app_users_email_lower_key` unique index.
**How to avoid:** Decide policy: either block re-invite with a clear message, or treat re-invite as "regenerate link" using `type:'recovery'`/`magiclink` for the existing user. Use `upsert`/`on conflict do nothing` on `app_users`, and catch the GoTrue "already registered" error to surface a meaningful message. [ASSUMED: pilot re-invite is rare — planner picks policy]

### Pitfall 5: Deactivation order (D-12) enforced only in UI
**What goes wrong:** A direct server-action call (or future code path) deactivates a company with active properties, leaving live `/pickup` links under a "removed" company.
**How to avoid:** Enforce "no active children" in the server action (count active children, return `t.deactivateCompanyBlocked`), and back hard-delete with FK `on delete restrict`. App-level check gives clean copy; the FK constraint is the integrity backstop.

### Pitfall 6: Service-role client accidentally imported into a client component
**What goes wrong:** Bundling the service-role key into the browser (full RLS bypass).
**How to avoid:** `admin.ts` already has `import "server-only"` (build fails on client import). Keep all invite/write logic in `"use server"` actions; never import `createAdminClient` from a `"use client"` file.

```sql
-- Target: Balkanity ref qyhdogajtmnvxphrslwm ONLY. NEVER Kalvia (utyatpadtibqqswsfvtr).

  id             uuid primary key default gen_random_uuid(),
  property_id    uuid not null references public.properties(id) on delete restrict,
  slug           text not null,
  label          text not null,
  address        text,
  zone           text,
  airport        text,
  price_cents    integer not null check (price_cents >= 0),       -- EUR minor units (D-07)
  commission_pct integer not null check (commission_pct between 0 and 100), -- whole % (D-05)
  active         boolean not null default true,
  created_at     timestamptz not null default now()
);

create unique index destinations_slug_key on public.destinations (slug);

-- Driver profile (D-02) — platform-generic, name + phone keyed to the auth user.
create table public.driver_profiles (
  user_id    uuid primary key references auth.users(id) on delete cascade,
  name       text not null,
  phone      text,
  created_at timestamptz not null default now()
);

-- RLS: enable on all four; admin-read policy; NO anon write policy (writes = service-role).
-- Mirror 0001: deny-by-default for anon/authenticated, explicit admin SELECT only.
alter table public.companies       enable row level security;
alter table public.properties      enable row level security;
alter table public.destinations    enable row level security;
alter table public.driver_profiles enable row level security;

-- Reusable admin predicate via the existing app_users role table.
create policy "companies_admin_read" on public.companies for select to authenticated
  using (exists (select 1 from public.app_users a
                 where a.id = (select auth.uid()) and a.role = 'admin'));
-- ...repeat the admin-read policy for properties, destinations, driver_profiles.
-- (Phase 4 will add a narrow anon/guest SELECT for active destination slugs — NOT this phase.)
```

### Known Threat Patterns for {Next.js App Router + Supabase}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Open redirect via attacker-controlled `redirectTo`/`Origin` | Tampering | Use trusted `NEXT_PUBLIC_SITE_URL`; allowlist redirect URLs (WR-04) |
| Service-role key leaking to client bundle | Information Disclosure | `import "server-only"` in `admin.ts` (build fails on client import); never `NEXT_PUBLIC_` the key |
| Privilege escalation (non-admin writing supply data) | Elevation of Privilege | Server-action re-gate via `getCurrentRole()` + admin-only RLS + no anon write policy |
| SQL injection | Tampering | supabase-js parameterized queries / RLS; never string-concatenate slugs/input into SQL |
| Account enumeration on re-invite | Information Disclosure | Generic messaging; do not reveal whether an email already has an account (mirror forgot-password) |
| Mass assignment via `app_metadata`/role tampering | Elevation of Privilege | Role written explicitly server-side as `'driver'`; never derive role from client input |
| Direct server-action invocation bypassing UI gates (D-12) | Tampering | Enforce lifecycle rules in the action + FK `on delete restrict`, not UI-only |
