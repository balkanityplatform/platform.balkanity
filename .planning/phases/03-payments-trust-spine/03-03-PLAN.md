<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client component -> platform/payments/* | A client import must fail the build (secret-key leak guard) |
| create-session input -> Stripe API | Untrusted `transfer_id` must resolve to a real row before any charge intent is created |
| request Origin header -> success/cancel URLs | Origin is attacker-controlled; URLs must come from a trusted constant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-ID2 | Information Disclosure / EoP | `STRIPE_SECRET_KEY` in `stripe.ts`/`fee.ts` | mitigate | `import "server-only"` first line on `stripe.ts`/`fee.ts` (build fails on client import, mirrors `admin.ts`); key read from non-public `STRIPE_SECRET_KEY`, never `NEXT_PUBLIC_` (asserted by the verify grep). |
| T-03-V5 | Tampering / Input Validation | `createCheckoutSession` input | mitigate | zod `z.string().uuid()` on `transfer_id` + resolve to a real `wp_transfers` row before calling Stripe (D-03); generic errors, no provider leak (Phase 2 boundary pattern). |
| T-03-WR | Spoofing / Tampering | `success_url`/`cancel_url` construction | mitigate | Build URLs from a trusted SITE_URL constant (`NEXT_PUBLIC_SITE_URL`), never the request Origin header (carried-forward WR-04 lock); the success page is display-only regardless (enforced in Plan 04). |
| T-03-MNY | Tampering | money precision at the Stripe boundary | mitigate | Pass integer `unit_amount` directly (cents from `wp_transfers.amount_cents`); record fee from `balance_transaction.fee` (authoritative integer cents) — no float math (Phase 2 integer-cents lock). |
</threat_model>
