<objective>
Build the booking vertical slice: a guest opens `/pickup/<slug>`, sees the destination + fare, fills the short guestless form (with the prepaid-&-non-refundable disclosure), and on submit a `requested` transfer row is created and they are 303-redirected to the Phase-3 Stripe Checkout. This is the thinnest end-to-end "guest can book and pay" slice — after this plan a real user can do something they could not before: book and pay for a transfer.

Purpose: Delivers BOOK-01 (slug page + fare), BOOK-02 (guestless form + server zod boundary), BOOK-03 (requested row + code-created Checkout Session), BOOK-04 (prepaid/non-refundable disclosure before payment). Reuses `createCheckoutSession` verbatim (no second paid writer; no client Stripe.js). The amount is server-trusted (re-read price by slug — Pitfall 5). Turns the Plan-01 `app/pickup/[slug]/booking.test.ts` RED spec GREEN.
Output: `/pickup/[slug]` page + form + action + the PaxStepper primitive.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| guest browser → `createBooking` server action (FormData) | Untrusted, unauthenticated public input crosses here; zod re-validates everything and the amount is server-re-read. |
| `createBooking` → `wp_transfers` (service-role insert) | Service-role bypasses RLS; the zod schema + the public-surface design (no role to check) are the only gates on the insert shape. |
| `createBooking` → Stripe (createCheckoutSession) | The charge amount must be server-sourced; a tampered FormData amount must never reach Stripe. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-TMP3 | Tampering | Client posts a tampered `amount`/`price` to underpay | mitigate | `createBooking` re-reads `destinations.price_cents` by slug server-side and uses it for both the row and the Session; the form submits no amount input (Task 1/3 acceptance greps enforce both). |
| T-04-V5 | Input Validation | Missing/garbage booking fields, past arrival, out-of-range pax | mitigate | zod `bookingSchema` at the trust boundary (D-01/02/03/04) with dictionary-keyed generic errors; a future-arrival check; pax bounded 1–8 (mirrored by the 0004 `pax` CHECK constraint). |
| T-04-SPOOF | Spoofing/Tampering | A second `paid` writer sneaks into the booking path | mitigate | The action writes only `status:'requested'`; `single-writer.test.ts` must stay green (Task 1 acceptance). The webhook remains the sole `paid` writer. |
| T-04-ID4 | Information Disclosure | Inactive destination details leak via `/pickup/<inactive-slug>` | mitigate | The page reads `active` and renders the neutral unavailable state for `active=false`; the 0004 anon policy also returns zero rows for inactive slugs. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan or phase (RESEARCH §Package Legitimacy Audit: zero new packages). No slopcheck required. |
</threat_model>
