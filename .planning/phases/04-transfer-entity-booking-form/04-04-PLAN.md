<objective>
Build the tracking + confirmation vertical slice: after the verified webhook flips a transfer to `paid`, a booking-confirmation magic link is generated (send stubbed ‚Üí Phase 7); the guest clicks it, the existing `/auth/confirm` route establishes a guest session and lands them on `/status/<id>`; the status page reads ONLY their own transfer via RLS and renders the live 8-state lifecycle timeline + a "Paid ‚ā¨X on {date}" receipt (+ the post-claim driver reveal); and `/track` lets them re-request a fresh link by email. After this plan a real user can do something new: track their transfer end-to-end and recover access by email.

Turns the Plan-01 confirmation + guest-status specs GREEN.
Output: confirmation stub, LifecycleTimeline, status + track routes, webhook/auth-confirm/sw/pay-success modifications.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stripe webhook (verified) ‚Üí confirmation email send | The email hangs off the already-verified paid transition; a send failure must not affect the money write. |
| magic-link click ‚Üí `/auth/confirm` (`next` param) | The `next` param is attacker-controllable; it must be allowlisted before use (open-redirect boundary). |
| guest cookie session ‚Üí `/status/<id>` (RLS read) | The guest JWT email is the only key; RLS decides which row (if any) is returned. |
| status page ‚Üí `driver_profiles` (service-role driver reveal) | A service-role read bypasses RLS; it must be narrowed to only {name, phone} and gated on the already-RLS-authorized owning transfer row so it never widens the guest's reach. |
| `/track` email input ‚Üí generateLink | Account-existence must not be revealed in the response. |
| SW cache ‚Üí guest documents | A stale-cached status/PII page is a correctness/security hazard. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-TMP4 | Tampering | Open redirect via the magic-link `next` param | mitigate | `/auth/confirm` allowlists `next` to `^/status/[0-9a-f-]{36}$`, defaults `/`, never forwards verbatim (WR-03); applied in both auth branches (Task 3 acceptance). |
| T-04-ID5 | Information Disclosure | Guest reads another guest's transfer/PII on `/status` | mitigate | RLS guest-self-read (`auth.jwt() email = guest_email`) is the boundary; the page uses the cookie/anon client + `auth.getUser()` (never getSession) for the transfer read, no service-role read of the guest row (Task 2 acceptance). |
