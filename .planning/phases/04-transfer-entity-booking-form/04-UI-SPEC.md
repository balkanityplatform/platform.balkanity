# Phase 4 — UI Design Contract

> Visual and interaction contract for the **guest-facing mobile PWA** surfaces Phase 4 introduces: the `/pickup/<slug>` booking page, the booking form + prepaid/non-refundable disclosure, the Stripe Checkout hand-off, the passwordless magic-link status page (lifecycle timeline + payment receipt + post-claim driver reveal), and the `/track` re-access page. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inheritance:** Phases 1–2 locked the platform design system (PLAT-03) and the admin-console contract. This spec reuses those tokens and primitives **verbatim** and adds only what is new to the guest surface. Do not re-derive tokens; consume them. The one surface shift: Phase 2 was the slate **admin desktop console**; Phase 4 is the warm **guest mobile PWA** — same palette, same primitives, mobile-first layout.

## Spacing Scale

Declared values (multiples of 4 only — matches the px values already hard-coded in the Phase 1/2 primitives). Reused verbatim; no new tokens.

| Token | Value | Usage in Phase 4 |
|-------|-------|------------------|
| xs | 4px | Dot↔label gap (StatusDot), tight label/help-text gap, timeline dot↔line gap |
| sm | 8px | Field internal vertical rhythm, stepper button↔value gap, receipt line gap |
| md | 16px | Default gap between form fields; input horizontal padding (`px-[16px]`); page body side padding (mobile) |
| lg | 24px | Card padding (`p-[24px]`, locked); Button horizontal padding (`px-[24px]`, locked); gap between form field groups |
| xl | 32px | Gap between major sections (fare summary ↔ form ↔ disclosure ↔ submit); top spacing under the page header |
| 2xl | 48px | Major surface breaks on the status page (timeline ↔ receipt ↔ driver-contact block) |
| 3xl | 64px | Reserved for wide-viewport outer spacing; mobile uses md/lg side padding |

Exceptions:
- CTA height is a fixed **52px** (brand-locked, off the 4-scale) with a **44px** minimum hit-target floor (WCAG 2.5.5 / mobile PWA). The same 52px geometry applies to `TextField` inputs.
- **All** interactive guest controls (inputs, the pax/luggage stepper `+`/`−` buttons, the disclosure checkbox, the "track my booking" submit, magic-link CTAs) must be **≥44px** tall/wide — this is the guest mobile-PWA touch-target contract and is non-negotiable for this phase.

## Typography

Reuse the Phase 1/2 roles exactly. Four sizes, two weights (regular 400 + semibold 600). No new sizes, no new weights.

| Role | Size | Weight | Line Height | Usage in Phase 4 |
|------|------|--------|-------------|------------------|
| Body | 16px | 400 | 1.5 | Field values, help text, disclosure body copy, receipt values, trip-summary values, timeline timestamps |
| Label | 14px | 600 | 1.4 | Field labels, StatusDot labels, timeline state labels, receipt line labels, fare-summary captions |
| Heading | 20px | 600 | 1.2 | Card titles ("Your trip", "Payment", "Booking status"), section headings |
| Display | 28px | 600 | 1.2 | Page title on `/pickup/<slug>` (destination label) and the status-page title |

Notes:
- Weights are strictly **400** and **600** — no 500/700.
- The fare amount on `/pickup/<slug>` and the "Paid €X" amount on the receipt are emphasised by **weight (600)** and the `slate` colour — NOT by a new size or the accent colour.
- Body line-height 1.5; heading/display 1.2; the 14px label uses `leading-[1.4]` (matches StatusDot / TextField).
- Inputs render guest-typed values at **16px** (prevents iOS Safari zoom-on-focus — important for the mobile PWA).

## Color

The locked six-colour-plus-white palette (`app/globals.css` `@theme`). Phase 4 maps it to the 60/30/10 model for the **guest mobile PWA** surface (warm/light, not the slate admin console).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` white | Page background, form surface, card surface — the warm-light guest base |
| Secondary (30%) | `#2f4858` slate | Headings, body text, field labels, the fare/receipt amounts, page chrome |
| Accent (10%) | `#029b87` teal | See reserved-for list below |
| Destructive / error | `#e44b4b` coral | Validation errors, the `cancelled` state dot+label, non-refundable emphasis word, error states |

Supporting brand tokens used **semantically via StatusDot only** (NOT as additional interactive accents):
- `#047982` teal2 — `paid` state dot (and the receipt's "Paid" emphasis is slate weight, not teal2 fill).
- `#febe21` amber — `en_route` / `arrived` state dots; the "live/attention" cue on the active timeline row.
- `#66676f` grey — muted help text, timestamps, future/not-yet-reached timeline rows, `requested` / `completed` state dots.

**Accent (teal) reserved for — explicit list, never "all interactive elements":**
- The primary CTA fill (`Button` primary): **"Continue to payment"** on the booking form, **"Send my booking link"** on `/track`.
- The active/focused form-field border + the `focus-visible` outline (already in `TextField`/`Button`).
- The active stepper button affordance and the checked state of the disclosure acknowledgement checkbox.
- Text links (e.g. "track my booking" link, the magic-link CTA inside the status prompt).

Accent is NOT used for: every button (the "Back"/secondary action uses the `ghost` outline-slate variant), card borders, body text, the fare/receipt amount, decorative fills, or the timeline connector line (that is `grey/30`).

**Status & lifecycle (reuse StatusDot map verbatim — colour is never the sole signal, WCAG 1.4.1):**
- `requested` → grey dot + "Requested"
- `paid` → teal2 dot + "Paid"
- `claimed` → teal dot + "Claimed"
- `en_route` → amber dot + "En route"
- `arrived` → amber dot + "Arrived"
- `picked_up` → teal dot + "Picked up"
- `completed` → grey dot + "Completed"
- `cancelled` → coral dot + "Cancelled"

The lifecycle timeline (SC4) renders a dot **and** a text label for **every** state including `en_route`; the current state row is emphasised (label weight 600 + a subtle slate-on-white treatment), reached-but-past rows are full-opacity, and not-yet-reached rows are muted (`grey`, reduced opacity) — but the label text is always present so colour is never load-bearing.

### Interaction contracts

- **Server-trusted amount (Pitfall 5):** the displayed `€{amount}` is the destination's `price_cents`; the form NEVER submits the amount — the server re-reads it. The UI shows it read-only.
- **Disclosure gate (BOOK-04):** the "Continue to payment" CTA is enabled only when the non-refundable acknowledgement checkbox is checked; an unchecked submit shows the blocked-submit error inline. The disclosure is visible above the CTA, not behind a link.
- **Pax stepper (D-03):** `−` disabled at 1, `+` disabled at 8; value is keyboard-operable and announced; a hidden `pax` input carries the value to the server action. Luggage stepper identical with `min=0` and no shown ceiling.
- **Checkout hand-off (BOOK-03):** on valid submit the server inserts the `requested` row and 303-redirects to the hosted Stripe Checkout — no in-app payment UI; the form shows a pending/disabled state on submit to prevent double-submit.
- **Success page (SC5):** `/pay/success` is display-only and NEVER renders "paid"/receipt as authoritative — it shows a neutral "Payment received — we're confirming it" state and points to the status link; the verified webhook remains the sole `paid` writer.
- **Status liveness (Claude's discretion / roadmap lock):** the status page fetches fresh on load (NetworkFirst — never served stale from the SW cache); a light client poll is the only acceptable "live" upgrade — no Realtime required for the pilot.
- **Timeline (SC4):** renders all 8 states top-to-bottom in lifecycle order; each row is dot+label; the current state is emphasised; `cancelled` is rendered as a distinct terminal row (coral) only when reached — it is not shown inline in the happy-path order.
- **Magic-link landing (AUTH-02):** clicking the emailed link lands the guest on `/status/<id>` via the existing `/auth/confirm` route; if the session is missing/expired the page shows the session-expired state with the "Get a new link" CTA routing to `/track` (no open-redirect; `next` allowlisted to `/status/<uuid>`).
