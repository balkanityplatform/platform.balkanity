<objective>
Build the driver pool vertical slice end-to-end: open `/driver` (driver-guarded) → read the masked `wp_pool()` → render limited-detail cards → tap Claim → on a win land on the transfer detail (full PII, rendered from the RPC's returned row), on a loss show a neutral toast and remove the card. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| driver browser→/driver RSC | Untrusted authenticated driver; role re-gated server-side via getCurrentRole(). |
| client→claimAction (Server Action) | The claim crosses to the server; identity derived from auth.uid() inside the RPC, never a client-supplied driver id. |
| SW cache→pool data | Stale claim state is an integrity risk; pool data must be NetworkFirst. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-PII | Information Disclosure | pool read | mitigate | Pool read is `wp_pool()` RPC (structural PII omission); no PII key selected into the payload (Pitfall 11). |
| T-06-RACE | Tampering | claim path | mitigate | claimAction calls `claim_transfer` (one atomic UPDATE) via the caller-auth wrapper only; no read-then-write, no pre-check (Pitfall 12). |
| T-06-CLIENT | Spoofing | claim identity | mitigate | Caller-auth client only; auth.uid() resolves the real driver inside the RPC; never service-role on the claim path. |
| T-06-STALE | Integrity | SW cache | mitigate | `/driver` pool data NetworkFirst / non-precached; graceful `already_claimed` is the second line of defence (Pitfall 4). |
| T-06-SC | Tampering | npm installs | accept | No package installs (zero new deps). |
</threat_model>
