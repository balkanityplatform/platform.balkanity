<objective>
Build the driver "My run" vertical slice: `/driver/run` lists the driver's own active claimed transfers ordered by arrival time, each card carries a compact read-only `LifecycleTimeline` plus an inline single next-step 52px CTA that advances the one next lifecycle edge in place, and reaching `completed` drops the card into a "Completed today" collapsed section. `/driver/run/[id]` shows the full post-claim detail (PII). The advance write is a NEW gated service-role Server Action with an ownership check (the driver write path that did not exist before). After this slice a driver can work a claim from `claimed` all the way to `completed`.

Purpose: CLAIM-05 (advance status from My run), CLAIM-06 (run ordered by arrival), and the CLAIM-04 "cannot un-claim" guarantee land here. The driver status-advance write is a gated service-role Server Action per D-13 (NOT a new RLS write policy — that would re-open Phase 5's closed no-write surface — and NOT a client-side write): the action re-checks `getCurrentRole()` + ownership before writing, and the migration-0004 trigger stays the state-legality backstop. This is the one genuinely-new write path in the driver surface — Pitfall 1 (no driver RLS write policy) is resolved exactly this way.

Output: `app/driver/run/page.tsx`, `app/driver/run/RunView.tsx`, `app/driver/run/[id]/page.tsx`, and the `advanceStatus` action appended to `app/driver/actions.ts`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| driver browser→advanceStatus | Untrusted; the action re-gates role AND verifies row ownership before any write. |
| service-role write→wp_transfers | service-role bypasses RLS, so the in-action ownership check is the only authz gate; the trigger is the state-legality backstop. |
| run read→claiming-driver RLS | The run/detail reads rely on RLS to scope rows to the owning driver — never a broad service-role read into the UI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-EOP1 | Elevation of Privilege | advanceStatus | mitigate | `driver_id === auth.uid()` ownership check + `getCurrentRole()==='driver'` gate; a forged call for another driver's transfer is rejected (CLAIM-04, D-13). |
| T-06-LEGAL | Tampering | status write | mitigate | Next edge resolved via ALLOWED_TRANSITIONS (forward only, never cancelled/paid); migration-0004 trigger rejects any illegal jump regardless. |
| T-06-CONCUR | Tampering | double-advance | mitigate | `.eq("status", current)` optimistic guard — a stale double-advance affects 0 rows. |
| T-06-PII | Information Disclosure | run/detail read | mitigate | Reads on the caller-auth client; claiming-driver RLS scopes rows to the owner; full PII is legitimate only post-claim. |
| T-06-SC | Tampering | npm installs | accept | No package installs (zero new deps). |
</threat_model>
