# Phase 7 ŌĆö UI Design Contract

> Visual and interaction contract for Phase 7 (Notifications). The frontend surface is intentionally small and **reuses the locked Phase 1 design system** (`app/globals.css` `@theme`, Montserrat, the `platform/ui/` primitives). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Two new presentational surfaces only:**
> 1. **In-app notification bell + feed** shown to **Drivers AND Admin** (NOT guests) ŌĆö unread count badge, opening the feed/item marks read, "mark all read" action (CONTEXT D-01/D-05). Updates via **poll-on-focus + ~30ŌĆō60s interval, NOT Realtime** (D-04 ŌĆö reuses the proven `app/driver/PoolView.tsx` poll shape).
> 2. **Driver daily-digest preference UI** ŌĆö a per-driver opt-in `Toggle` (off by default) + a time-of-day `Select` for the self-chosen send hour (D-07/D-08).
>
> **Transactional email templates** (guest confirmation/assigned/arrived, admin alert, driver invite, daily digest) are server-rendered HTML built via the existing `getDict()` token-interpolation pattern (`platform/transfers/confirmation-email.ts`). They are NOT React/PWA UI ŌĆö only the **minimal email layout/brand contract** in ┬¦"Email Layout Contract" applies to them.

**Semantic colours (status-driven, never decorative) ŌĆö locked brand map:**
- `--coral #E44B4B` ŌåÆ **the admin `email_cap_near` notification only** (D-11) ŌĆö the one alarm-tier in-app notification; rendered with a coral unread dot **plus** a text marker ("Email cap nearing"), never coral alone. **Color-is-never-the-sole-signal (WCAG 1.4.1, brand-locked):** unread state is shown by a **teal dot + 600-weight title** (two cues), never by colour alone. The cap-near alarm carries a coral dot **and** an explicit text label.

## Bell Affordance Decision (resolves the no-icon constraint)

The brand pictogram set has **no bell glyph**, and the locked rule is *never invent or re-draw icons* (pictograms README). Therefore:

- The bell trigger is a **labelled text control** reading **"Alerts"** (EN) / **"ąśąĘą▓ąĄčüčéąĖčÅ"** (BG), with the unread count rendered as a teal numeric badge beside the label.
- It is a native `<button>` (or the `Button` primitive in a compact variant) with `aria-label` = "Alerts, {n} unread" so screen readers announce the count.
- Do **NOT** add an SVG bell, a Unicode ­¤öö, or any emoji ŌĆö that would either invent a mark (forbidden) or violate the no-random-emoji rule. A text label is the brand-consistent affordance.
- If a future phase ships a real Balkanity bell pictogram, it may replace the text label; until then "Alerts" + count is the contract.

## Email Layout Contract (minimal ŌĆö server-rendered HTML, NOT PWA UI)

Templates are plain HTML strings via the existing `getDict()` token-interpolation pattern (`confirmation-email.ts`); do NOT add `react-email`. The contract is intentionally light ŌĆö single-column transactional mail.

| Rule | Value |
|------|-------|
| Sender | `noreply@send.balkanity.com` (D-13, verified subdomain ŌĆö phase-completion gate) |
| Language | Guest emails honour `wp_transfers.locale` (EN/BG, EN fallback when NULL, D-17); admin + invite EN |
| Layout | Single column, max-width ~560px, left-aligned; safe system font fallback (`Montserrat, Arial, sans-serif`) ŌĆö webfonts are unreliable in email clients |
| Brand colour | One teal (`#029B87`) accent (heading rule / single CTA button); body text slate (`#2F4858`) on white; no multi-colour decoration |
| CTA | At most ONE button per email (e.g. invite "Set your password", confirmation "Track your transfer"); teal fill, white text, Ōēź44px tall |
| Body type | ~16px body, ~20ŌĆō24px heading, line-height 1.5 (inline styles ŌĆö email clients ignore `<style>` blocks unevenly) |
| PII boundary | The "driver assigned" email is the ONLY place driver first name + phone appears, and ONLY to the guest (`to:` is always the row's `guest_email`); never in admin/driver-facing mail (D-16, Pitfall 5) |
| Footer | Plain text sender identity + a one-line "transactional email" note; no marketing, no tracking pixels |

## Interaction Contracts (phase-specific behaviour the executor must honour)

| Contract | Rule | Source |
|----------|------|--------|
| Bell "live" refresh | Refetch own notifications on `focus` + `visibilitychange` + a ~30ŌĆō60s interval poll while visible. **NOT Supabase Realtime** (overrides ROADMAP SC#1 "via Realtime" wording). Copy the `app/driver/PoolView.tsx` poll shape verbatim. | D-04, RESEARCH Pattern 2 / Pitfall 1 |
| Unread badge | Count = own `notifications` rows where `read_at IS NULL`, read RSC-side under caller-auth RLS; badge hidden at 0; show "9+" past 9. | D-05 |
| Mark read on open | Opening the feed (or an item) marks the relevant items read via a **gated service-role `markRead` server action** (caller-identity check `recipient_id === auth.uid()`, then service-role UPDATE) ŌĆö NOT a client write RLS policy (preserves the no-write-policy lock). | D-05, RESEARCH Pattern 1 / Open Q1 |
| Mark all read | "Mark all read" CTA ŌåÆ gated service-role `markAllRead` action over the caller's own unread rows. | D-05 |
| Feed ordering | Newest first (`created_at desc`); unread items visually lead (teal dot + 600 title); read items grey/400. | D-05 |
| Digest opt-in default | `Toggle` is **OFF by default** (opt-in, protects the Resend cap). Time `Select` is disabled/ignored until the toggle is on. | D-07/D-08 |
| Digest send hour | `Select` of whole-hour options (the driver's self-chosen local send hour); stored as a driver preference. **The time trigger that fires due digests is Phase 8 (Supabase cron) ŌĆö this UI only sets the preference + an invokable send exists.** | D-08 (seam flagged) |
| No guest bell | Guests get NO in-app bell ŌĆö the guest status page already shows their single transfer's state (out of scope). | D-01 |
| Failure tone | A lost poll / failed mark-read is a transport issue ŌåÆ `Toast` error tone (coral) only on genuine failure; routine "no new alerts" is the neutral empty state, never an error. | Phase 6 Toast convention |
