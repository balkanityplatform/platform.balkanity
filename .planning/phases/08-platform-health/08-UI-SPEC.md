# Phase 8 — UI Design Contract

> Visual and interaction contract for the Platform Health admin surfaces. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** Phase 8 (Platform Health) is almost entirely backend wiring (pg_cron + pg_net + route handlers + a `health_events` table). The ONLY user-observable UI is **admin-console-only** (D-05, D-06): an **email-cap gauge** (HLTH-03), a **stuck-transfer alert panel** (HLTH-04), a **reconciliation / health-events list** (HLTH-02), and the existing **NotificationBell** carrying the new alert kinds. There is **no guest UI and no driver UI** in this phase. Every surface inherits the already-locked manual design system (PLAT-03) and the slate-console chrome established in Phases 1–7 — this contract pins those existing tokens; it does NOT introduce a new design language.

## Spacing Scale

Established 4-point scale used verbatim across the console (Tailwind v4 arbitrary-value `[Npx]` utilities, as seen in `app/admin/page.tsx` and `TransfersView.tsx`). Phase 8 reuses these exactly:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (StatusDot), badge inner padding (`py-[2px]` / `[4px]`) |
| sm | 8px | Chip gaps, inline element spacing, logo chip padding |
| md | 16px | Default element spacing, header `py`, control padding, list item `px` |
| lg | 24px | Section padding (`px-[24px]`), header horizontal padding, Card padding, vertical stack gaps |
| xl | 32px | Nav block / sub-section top margin (`mt-[32px]`) |
| 2xl | 48px | Main content vertical breathing room (`py-[48px]`) |
| 3xl | 64px | Reserved page-level spacing (not yet needed on health widgets) |

**Exceptions (established, carried forward — not new):**
- **44px minimum hit target** on all interactive controls (links, filter chips, buttons) — brand accessibility rule (`min-h-[44px]`). This is a touch-target floor, not a spacing-scale value.
- **52px primary CTA height** — brand rule for primary actions (Phase 1 SC-5). Phase 8 has few/no primary CTAs (health surfaces are read-only); if a "Replay webhook" or "Resolve" action ships, it uses the 52px primary CTA via `platform/ui/Button`.
- **List-row min-height 56px** (`min-h-[56px]`) — established list-row rhythm from `TransfersView`; the stuck-transfer list and health-events list reuse it.
- **10px / 12px** appear only inside StatusDot (dot diameter) and the coral badge (uppercase micro-label) respectively — component-internal, inherited verbatim, not general spacing tokens.

## Typography

Established 4-size / 2-weight system (used verbatim in `app/admin/page.tsx`, `TransfersView.tsx`, `StatusDot.tsx`). Weights are exactly two: regular 400 and semibold 600. Phase 8 reuses these — no new sizes or weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display / Page heading (h1) | 28px | 600 (semibold) | 1.2 (`leading-[1.2]`) |
| Heading (h2 / section / empty-state heading) | 20px | 600 (semibold) | 1.3 (`leading-[1.3]`) |
| Body | 16px | 400 (regular) | 1.5 (`leading-[1.5]`) |
| Label / meta / status label | 14px | 600 (semibold) | 1.4 (`leading-[1.4]`) |

**Micro-badge exception (inherited):** the coral attention/alert badge renders at 12px / 600 uppercase with `tracking-wide` (`text-[12px] font-semibold uppercase tracking-wide`) — component-internal marker, carried verbatim from `TransfersView`. The email-cap gauge state badge (warning/at-cap) reuses this same 12px uppercase badge style.

**Phase 8 typography assignments:**
- Health page / panel heading → 28px (page) or 20px (panel section heading).
- Gauge numeric readout ("72 / 100 sent today") → 16px body for the figure label; the count itself MAY render at 20px/600 for emphasis (still within the declared scale).
- Stuck-transfer row / health-events row primary line → 16px/600 slate; meta line → 14px/400 grey (matches `TransfersView` row anatomy).

## Color

Locked six-colour brand palette (`app/globals.css` `@theme`), applied with the established 60/30/10 split (white dominant content surface, slate console chrome, teal accent). Phase 8 introduces NO new colours — it assigns the existing tokens to the new health-state semantics, consistent with the locked lifecycle/status colour map.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (white) | Page background, content surfaces, cards, list backgrounds |
| Secondary (30%) | `#2f4858` (slate) | Console header chrome, dark surfaces, headings/text, active filter chips |
| Accent (10%) | `#029b87` (teal) | Focus rings (`outline-teal`), links/primary actions, primary CTA, "healthy/OK" gauge state |
| Destructive / Alert | `#e44b4b` (coral) | Reconciliation money-discrepancy alerts, at-cap gauge state, stuck-transfer left-border + badge, error states, destructive confirmations |

**Supporting brand tokens (already in palette, used for state semantics — not additional accents):**
- `#febe21` (amber) — **gauge "warning" state at ~90/day (D-07)** and "in-progress/attention" signals. Amber = approaching cap; coral = at/over cap.
- `#047982` (teal2) — `paid` status dot (inherited from StatusDot map; appears in reconciliation/stuck rows that show a transfer status).
- `#66676f` (grey) — muted meta text, borders (`border-grey/30`), `requested`/`completed` status dots, neutral/empty-state body copy.

**Accent reserved for (explicit list — never "all interactive elements"):**
- Focus-visible outline ring on every interactive control (`focus-visible:outline-teal`).
- Text links / navigational links to a transfer or webhook-event detail.
- The single primary CTA per surface, IF one ships (e.g. a "Replay via webhook" / "Mark resolved" action) — 52px teal `Button`.
- The **"healthy / OK"** gauge fill/indicator (below the warning threshold).

**Health-state colour semantics (the one genuinely new mapping this phase pins):**

| Health state | Token | Trigger |
|--------------|-------|---------|
| Email-cap gauge — OK | teal `#029b87` | sent today `<` (cap − 10), i.e. `< 80` |
| Email-cap gauge — Warning | amber `#febe21` | `>= cap − 10` (≥ 80) — approaching the ~90/day soft cap (D-07) |
| Email-cap gauge — At/over cap | coral `#e44b4b` | `>= cap` (≥ 90) — soft cap reached; best-effort sends start dropping |
| Stuck-transfer alert (HLTH-04) | coral `#e44b4b` | paid + unclaimed within 12h of arrival (D-04) — left-border `border-l-4 border-l-coral` + text badge |
| Reconciliation money discrepancy (HLTH-02) | coral `#e44b4b` | a Stripe-paid session with no matching paid transfer (D-01) — the most severe signal; also the only one that emails (D-09) |
| Resolved / cleared health event | grey `#66676f` | `resolved_at` is set — muted, de-emphasised |

**WCAG 1.4.1 — colour is NEVER the sole signal (locked brand rule, threat T-06-COLOR):** every coral/amber state MUST also carry a TEXT marker. The gauge shows the numeric "{sent} / {cap}" figure AND a worded state label ("Approaching cap", "At cap"). Stuck rows carry the `stuckBadge` text. Reconciliation rows carry the `reconciliationBadge` text. Mirror the existing `needsAttentionBadge` pattern in `TransfersView` (coral background, white uppercase 12px label).

## Copywriting Contract

All copy MUST be authored as EN + BG dictionary keys (`platform/i18n/dictionary` → `en.ts` / `bg.ts`) behind the tsc Dict-parity gate — no inline strings (established convention, every prior phase). The strings below are the EN source; the planner adds the BG parity keys.

| Element | Copy |
|---------|------|
| Page / panel heading | **"Platform health"** (`healthTitle`) |
| Email-cap gauge label | **"Emails sent today"** (`emailCapLabel`) — figure rendered as "{sent} / {cap}" |
| Email-cap gauge — warning state label | **"Approaching daily cap"** (`emailCapWarning`) |
| Email-cap gauge — at-cap state label | **"Daily cap reached — non-critical emails paused"** (`emailCapAtCap`) |
| Stuck-transfer panel heading | **"Stuck transfers"** (`stuckHeading`) |
| Stuck-transfer row badge | **"UNCLAIMED"** (`stuckBadge`) — coral uppercase text marker |
| Stuck-transfer empty state heading | **"No stuck transfers"** (`stuckEmptyHeading`) |
| Stuck-transfer empty state body | **"Every paid transfer has a driver or has time to spare. Nothing needs chasing right now."** (`stuckEmptyBody`) |
| Reconciliation panel heading | **"Payment reconciliation"** (`reconHeading`) |
| Reconciliation row badge | **"NEEDS REVIEW"** (`reconBadge`) — coral uppercase text marker |
| Reconciliation empty state heading | **"All payments reconciled"** (`reconEmptyHeading`) |
| Reconciliation empty state body | **"Every paid Stripe session has a matching paid transfer. No discrepancies to investigate."** (`reconEmptyBody`) |
| Email-cap gauge empty/zero state | **"No emails sent yet today."** (`emailCapZero`) |
| Primary CTA (only if a resolve action ships) | **"Mark resolved"** (`healthResolveCta`) — clears a health event after human-driven webhook replay |
| Error state (widget read failed) | **"Couldn't load health data. Refresh to try again."** (`healthLoadFailed`) — mirrors the existing `alertsLoadFailed` pattern |
| Destructive confirmation | **None in this phase.** The sweep is detect-and-alert ONLY (D-01); it never writes `paid` and offers no destructive UI. If a future "Mark resolved" action ships it is non-destructive (sets `resolved_at`, no data loss) and needs no confirmation dialog. |

**Copy tone notes:**
- Health alerts describe the FACT and the human's next step (e.g. reconciliation alert → "investigate / replay via webhook"), never raw guest PII (the critical reconciliation email and in-app alert carry transfer id / event id / amount only — Security Domain V7, mirrors digest non-PII discipline).
- Empty states are reassuring, not blank — "all reconciled / nothing stuck" confirms the system is healthy, which is itself the signal an operator wants on a quiet pilot day.

## Layout & Interaction Notes (phase-specific, inherited patterns)

These are not new tokens — they pin which established patterns the executor reuses so the health surfaces are visually indistinguishable from the rest of the admin console.

- **Chrome:** slate console header (`bg-slate px-[24px] py-[16px]`) with the white-chip Balkanity logo + `LanguageToggle` (+ `NotificationBell` on the landing) — identical to `app/admin/page.tsx` / `TransfersView.tsx`.
- **Content container:** `mx-auto max-w-2xl px-[24px] py-[48px]`, vertical `flex flex-col gap-[24px]`.
- **Placement (Open Q2 from RESEARCH — planner's call, both acceptable):** either fold the gauge + panels into the existing `app/admin/page.tsx` landing, or a dedicated `/admin/health` page reachable from the console section nav. Either way, reuse the same chrome + container.
- **Panels/cards:** group each widget in the `Card` primitive (`rounded-md border border-grey/30 bg-white p-[24px]`).
- **Lists (stuck + health-events):** reuse the `TransfersView` list anatomy — `divide-y divide-grey/20 rounded-md border`, `min-h-[56px]` rows, primary line 16px/600 slate + meta line 14px/400 grey, coral `border-l-4` + text badge for flagged rows, each row a focusable link to `/admin/transfers/<id>` (or the webhook-event context).
- **Status:** any transfer status shown reuses `StatusDot` (dot + label, locked colour map) — never re-colour or re-label.
- **NotificationBell:** the new alert kinds (stuck, cap-near, reconciliation) flow into the EXISTING bell (Phase 7) — no new bell UI; only new notification rows.
- **Gauge:** a simple horizontal progress/meter (white track, fill colour by state token) with the "{sent} / {cap}" figure + worded state label beside it — no chart library, no new dependency (Don't-Hand-Roll / read-only RSC widget per RESEARCH).
- **Read-only:** every health surface is read-only RSC behind the existing `getCurrentRole === 'admin'` gate. No client write path, no destructive control (D-01).
