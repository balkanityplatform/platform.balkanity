# Phase 10 ‚ÄĒ UI Design Contract

> Visual and interaction contract for the **Guest UI Rebuild** ‚ÄĒ the boarding-pass "Transfer Pass" re-skin of the guest-facing booking + status surface. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope (presentation-only):** This phase re-skins the EXISTING guest routes ‚ÄĒ `app/pickup/[slug]/page.tsx` (+ `BookingForm.tsx`), `app/pay/success/page.tsx`, `app/pay/cancel/page.tsx`, `app/status/[id]/page.tsx`, `app/track/page.tsx`. **NO** change to backend, schema, auth, RLS, payment path, form fields, validation, or the Stripe Checkout flow. The pay action drives the existing code-created Checkout-session flow unchanged; the client redirect NEVER sets `paid` (single-writer invariant preserved). Magic-link auth for the status page is preserved as-is.
>
> **Foundation:** This surface **consumes** the Phase 9 design system ‚ÄĒ it does NOT re-invent tokens or components. All tokens already live in `app/globals.css` `@theme`; the three shared components (`StatusDot`, `RouteMotif`, `LifecycleStepper`) ship from `platform/ui/`. Reuse them verbatim.
>
> **Source of truth:** the Stitch mockups (`guest_transfer_pass_booking/`, `guest_transfer_pass_status/`) + `Branding/.../balkanity_path/DESIGN.md`, **corrected** on contact. Brand primary is **`#029B87`** ‚ÄĒ the mockups' `#00685a`/`primary` token family is **rejected**. Material Symbols icons ‚Üí Phase 9 line pictograms + the committed brand Transfer Badge. The four locked layout decisions below are fixed.

## Locked Decisions (this phase)

These four decisions were answered by the user and are FIXED ‚ÄĒ do not re-litigate.

| # | Decision | Contract |
|---|----------|----------|
| 1 | **Pass metaphor = "Silhouette, drop barcode."** | Keep the boarding-pass look: teal (`#029B87`) header band, route header, perforated dashed divider + circular notch cutouts as **pure decoration**. **DROP the fake barcode** and the long fake digit string. Show the **real, truncated transfer ID** (not an invented "BK-2941-X" / "928374019283"). |
| 2 | **Pass + form = "Pass header frames form."** | The `/pickup/[slug]` booking screen is ONE stacked page: (a) route/fare **pass header** on top ‚ÄĒ airport‚Üíproperty via `RouteMotif`, total prepaid, payment-status row; (b) the **restyled input form** below (same fields/validation); (c) the **pay CTA + Stripe trust footer**. The pass frames the form; it does not replace it. |
| 3 | **Drop "Est. Pickup."** | Remove the invented "Est. Pickup" field entirely. The details grid shows ONLY real collected fields: arrival **Date**, **Flight No.**, **Guests** (pax), arrival **time**. |
| 4 | **Status page = "Drop extras, horizontal stepper."** | DROP the live GPS map, driver call/chat buttons, barcode / "View Travel Vouchers" CTA, and the admin nav shells (sidebar + top bar + bottom nav). Use the Phase 9 **horizontal `LifecycleStepper`** (DS-04) for the live lifecycle ‚ÄĒ NOT the mockup's vertical stepper. KEEP the existing post-claim driver reveal (first name + phone as **plain text** ‚ÄĒ no call/chat buttons) and the receipt/payment block. |

**Presentation-only omissions (ROADMAP v1.1 truthfulness rule):** live GPS map, "Estimated arrival: 8 mins" ETA, vehicle make/model/plate (Skoda Superb ‚ÄĘ TV-AB123), driver avatar initials, driver ratings, "Live Tracking" label, "View Travel Vouchers" / travel-voucher export, call + chat buttons. None have backing data ‚ÄĒ all omitted.

## Spacing Scale

Consumes the Phase 9 8px grid unchanged. All multiples of 4:

| Token | Value | Usage on this surface |
|-------|-------|-----------------------|
| xs | 4px | Icon‚ÜĒlabel gaps (StatusDot dot‚ÜĒlabel, details-grid icon‚ÜĒvalue) |
| sm | 8px | Compact stacking; standard 8px corner radius |
| md | 16px | Mobile gutter (`px-[16px]`); card gutter; default form-field vertical gap |
| lg | 24px | Pass-section padding; route-header vertical padding; between major page blocks |
| xl | 32px | Page top/bottom padding (`py-[48px]` retained where present); section breaks |
| 2xl | 48px | Page vertical rhythm; **input field height = 48px** |
| 3xl | 64px | Page-level spacing (rarely needed on this single-column surface) |

Exceptions (brand-locked, all ‚Č• multiple-of-4):
- **44px** minimum touch target on every interactive element (checkbox row, links, CTAs). WCAG 2.5.5; `--spacing-touch-target`.
- **52px** primary pay CTA height (`Button` `h-cta-height`; `--spacing-cta-height`).
- **10px** status dot diameter (decorative; from `StatusDot`).
- Page max width **480px** mobile-first single column (existing guest convention ‚ÄĒ `max-w-[480px]`). The pass card spans the full content column.

Margins: 16px mobile gutter. This surface is mobile-PWA-first; no desktop sidebar/top-bar (Decision 4 drops the admin shells).

## Typography

Montserrat exclusively, **400 + 600 pair** (the loaded weights). Phase 9 roles, consumed verbatim. The mockups' 700/800 and 40px `headline-xl` are NOT introduced ‚ÄĒ the airport-code "TIV/BVB" giant display is rendered at the **Display** role (28px/600), not 40px/700.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Display | 28px | 600 | 1.2 (34px) | Pass route endpoints / destination label / page title |
| Heading | 24px | 600 | 1.33 (32px) | Total prepaid amount; section headings ("Your trip", "Your details") |
| Body | 16px | 400 | 1.5 (24px) | Form values, receipt line, driver line, disclosure body |
| Label | 14px | 600 | 1.4 (‚Čą20px) | Field labels, details-grid captions (UPPERCASE permitted), status labels, stepper labels, "Transfer Pass" eyebrow |

Caption micro-text from the mockup (`text-[10px]`, `text-[11px]`) is **lifted to 14px Label** (or 12px `label-md` minimum) ‚ÄĒ do not ship sub-14px body text. UPPERCASE tracking is allowed on the eyebrow ("TRANSFER PASS") and grid captions ("DATE", "FLIGHT NO.", "GUESTS", "PAYMENT STATUS", "TOTAL PREPAID") as a styling treatment of the Label role.

## Color

The guest surface is the **warm light** variant (white base). Slate is used only for the pass header band text and chrome; the teal header band is the one large accent surface and is the pass's signature ‚ÄĒ counted as branded chrome, not arbitrary decoration.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` white | Page + pass-card background; form-field fill |
| Secondary (30%) | `#2F4858` slate | Headings, body text, field values, captions; the status-pass dark header band (status mockup uses a slate header ‚ÄĒ permitted) |
| Accent (10%) | `#029B87` teal | See reserved-for list below |
| Destructive | `#E44B4B` coral | Inline form validation errors, the unchecked-acknowledgement blocked error |

**Accent (`#029B87` teal) reserved for:** the booking pass **teal header band**, the primary **pay CTA** fill, the Stripe-trust **lock icon**, **links** (track / status / re-access CTAs, teal underline), the input **focus ring** (2px teal `focus-visible`), the `RouteMotif` **Transfer Badge** circle, and the `LifecycleStepper` **completed-step** teal circle. NOT for arbitrary decoration, not on every interactive element.

**Status / lifecycle colours** are owned by the shared components (do not re-map):
- Payment-status row on the booking pass uses `StatusDot` semantics: pre-payment shows **amber** ("Pending prepayment" ‚ÄĒ `requested`/unpaid), teal2 once `paid`. Always coloured dot **+ worded label** (WCAG 1.4.1).
- The status-page `LifecycleStepper` encodes completed (teal + check) / active (amber) / pending (grey outline ring) by **shape**, never colour alone; each step keeps its worded label.

**Elevation:** tonal, not heavy. The mockup's `pass-shadow` (0 10px 25px black/20%) and `drop-shadow` are **softened** to the Phase 9 ambient shadow `0 4px 12px rgba(47,72,88,0.08)` or a 1px `#66676F`@20% border. The perforated divider is a 2px dashed line in `#66676F`@~30% (mockup used `#bcc9c5` ‚ÄĒ re-tinted to the brand grey). Notch cutouts are decorative circles matching the page background.

**Radii** (from `@theme`): pass card / large containers `rounded-lg` (16px) or `rounded-xl` (24px) for the boarding-pass feel; inputs/buttons `rounded` (8px) ‚Üí CTA may use `rounded-md`/`rounded-lg` to match the mockup's pill-ish button; Transfer Badge + status dots `rounded-full`.

| Element | Copy |
|---------|------|
| Primary CTA (booking pay) | **"Pay ‚ā¨{amount} & confirm"** ‚ÄĒ the existing `createBooking` continue action, restyled. (Existing `bookingContinueCta` may be re-worded to this; pending label `bookingContinuePending` = "Confirming‚Ä¶".)
