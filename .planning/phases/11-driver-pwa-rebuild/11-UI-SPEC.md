# Phase 11 — UI Design Contract

> Visual and interaction contract for the **Driver PWA Rebuild** — re-skinning the driver-facing surface to the mockup identity: Available transfers as claim cards (no pre-claim PII), a bottom navigation bar, a My Trips list, and an en-route trip detail with the shared progress stepper + Confirm-Arrival CTA. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope (presentation-only):** This phase re-skins the EXISTING driver routes — `app/driver/page.tsx` (+ `PoolView.tsx`), `app/driver/run/page.tsx` (+ `RunView.tsx`), `app/driver/run/[id]/page.tsx`, `app/driver/settings/page.tsx` (the de-facto Profile). **NO** change to backend, schema, auth, RLS, payment path, the masked `wp_pool()` read, the atomic `claim_transfer` RPC, or the gated `advanceStatus` write. Claim cards render ONLY the 9 non-PII columns the `wp_pool()` RPC returns — masking is structural at the data layer (migration 0005, Pitfall 11); this surface merely renders what the RPC hands it. Claim still calls `claimAction(id)` (first-to-claim-wins / already-claimed semantics unchanged, CLAIM-04). Confirm-Arrival is the existing `advanceStatus(id)` gated service-role write (D-13). Driver warm-light chrome (white base), ≥44px hit targets, 52px primary CTAs.
>
> **Foundation:** This surface **consumes** the Phase 9 design system and follows the Phase 10 surface-local pattern — it does NOT re-invent tokens or shared components. All tokens already live in `app/globals.css` `@theme`; the shared components ship from `platform/ui/` (`StatusDot`, `RouteMotif`, `LifecycleStepper`). New driver-only pieces (claim card, bottom nav, trip card) live under the driver routes (mirroring `app/(guest)/_pass/`), NOT in `platform/ui/`.
>
> **Source of truth:** the Stitch mockups (driver Available / My Trips / trip-detail screens) + `Branding/.../balkanity_path/DESIGN.md`, **corrected** on contact. Brand primary is **`#029B87`** — the mockups' `#00685a`/`primary` token family is **rejected**. Material Symbols icons → Phase 9/10 line pictograms + the committed brand Transfer Badge. The locked decisions below are fixed.

## Locked Decisions (this phase)

These decisions are driven by the ROADMAP Phase 11 contract + the v1.1 truthfulness rule, and are FIXED — do not re-litigate.

| # | Decision | Contract |
|---|----------|----------|
| 1 | **Omit backend-less mockup features.** | DROP the live GPS map / route tracking, driver ratings (e.g. 4.9★), and the earnings dashboard (totals, +12% trends). None have backing data (ROADMAP Out of Scope). The trip detail shows NO map; My Trips shows NO earnings or ratings. |
| 2 | **Pre-claim cards show ZERO guest PII.** | The Available claim card renders ONLY the 9 masked `wp_pool()` columns: arrival date + pickup time, airport, zone (area only), flight_no (operational, non-PII per D-02), fare, pax, luggage. NO guest name / phone / exact address / notes anywhere pre-claim. This is the existing data-layer boundary surfaced, never re-implemented in UI. |
| 3 | **Bottom navigation = Available / My Trips / Profile.** | A persistent bottom nav bar replaces the current top-right-only nav. Three tabs: **Available** (`/driver`), **My Trips** (`/driver/run`), **Profile** (`/driver/settings`). Active tab highlighted in teal. ≥44px hit targets. The Alerts bell + Language toggle stay in a slim top header. |
| 4 | **En-route detail uses the horizontal `LifecycleStepper`.** | The trip detail (`/driver/run/[id]`) SWAPS the current vertical `LifecycleTimeline` for the Phase 9 horizontal `LifecycleStepper current={status}` (DS-04). It renders claimed passenger info, a `RouteMotif` route card, the passenger note, and a single **Confirm-Arrival** primary CTA wired to the existing `advanceStatus`. No map, no call/chat buttons, no vehicle details. |
| 5 | **Confirm-Arrival is the detail-page advance CTA; My-run inline advance is preserved.** | DUI-04 names "Confirm-Arrival" specifically — on the detail page the next-forward-edge advance CTA is shown with the lifecycle-appropriate label (en_route→"Confirm arrival"). The existing `RunView` inline advance CTAs stay functional (CLAIM-05/06). No un-claim / give-back control anywhere (CLAIM-04). |

**Presentation-only omissions (ROADMAP v1.1 truthfulness rule):** live GPS map, route-line tracking, ETA ("Estimated arrival: N mins"), vehicle make/model/plate, driver avatar/initials, driver ratings (★), earnings totals + trend %, call + chat affordances. None have backing data — all omitted.

## Spacing Scale

Consumes the Phase 9 8px grid unchanged. All multiples of 4:

| Token | Value | Usage on this surface |
|-------|-------|-----------------------|
| xs | 4px | Icon↔label gaps (StatusDot dot↔label, card meta-row icon↔value) |
| sm | 8px | Compact stacking; standard 8px corner radius; nav-icon↔label gap |
| md | 16px | Mobile gutter (`px-[16px]`/`px-[24px]` retained where present); card padding; card-to-card gap |
| lg | 24px | Card internal vertical rhythm; section padding; header/page horizontal padding |
| xl | 32px | Section breaks; empty-state vertical padding (`py-[32px]`) |
| 2xl | 48px | Major vertical rhythm |
| 3xl | 64px | Page-level spacing (rarely needed on this single-column surface) |

Exceptions (brand-locked, all ≥ multiple-of-4):
- **44px** minimum touch target on every interactive element — claim CTA, bottom-nav tabs, details links, advance CTA. WCAG 2.5.5; `--spacing-touch-target`.
- **52px** primary CTA height (Claim, Confirm-Arrival / advance) — `Button` `h-cta-height`; `--spacing-cta-height`. (ROADMAP: "52px primary CTAs".)
- **10px** status dot diameter (decorative; from `StatusDot`).
- **Bottom-nav bar height ≈ 64px** plus the device safe-area inset (`pb-[env(safe-area-inset-bottom)]`) — the nav is fixed to the viewport bottom; page content gets matching bottom padding so the last card is never occluded.
- Page content column: existing driver convention is `max-w-2xl` centered — retained (this surface is mobile-PWA-first; the claim/trip cards are full content-width).

Margins: 16–24px horizontal gutter (existing `px-[24px]` header/section padding retained).

## Typography

Montserrat exclusively, **400 + 600 pair** (the loaded weights). Phase 9 roles, consumed verbatim. The mockups' 700/800 and 40px display are NOT introduced.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Display | 28px | 600 | 1.2 (34px) | Page title ("My Trips"); large fare on the trip detail (optional) |
| Heading | 24px | 600 | 1.33 (32px) | Section headings; empty-state heading (mockup 20px → lifted toward Heading; existing 20px empty headings acceptable as `headline`-adjacent) |
| Body | 16px | 400 | 1.5 (24px) | Card primary line (arrival date·time), passenger info values, note body, fare value |
| Label | 14px | 600 | 1.4 (≈20px) | Card meta captions, status labels, stepper labels, nav-tab labels, "Unclaimed" badge, details-grid captions (UPPERCASE permitted) |

Caption micro-text from the mockup (`text-[10px]`/`text-[11px]`) is **lifted to 14px Label** (12px `label-md` absolute minimum) — do not ship sub-14px body text. Nav-tab labels render at 12px Label minimum (the one place 12px is acceptable, matching native bottom-nav convention); icons carry the redundant signal so the label is never the sole cue. UPPERCASE tracking is permitted on grid captions ("ARRIVAL", "FLIGHT", "FARE", "PASSENGERS", "LUGGAGE") and the "Unclaimed" badge as a styling treatment of the Label role.

## Color

The driver surface is the **warm light** variant (white base) — same as the existing driver chrome. No slate console here.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` white | Page + card background; bottom-nav bar background; header background |
| Secondary (30%) | `#2F4858` slate | Headings, body text, card values, captions; active nav-tab label/icon may darken to slate when not the live route |
| Accent (10%) | `#029B87` teal | See reserved-for list below |
| Destructive | `#E44B4B` coral | "Unclaimed" status badge, Cancelled status (hollow ring), claim-failure error toast |

**Accent (`#029B87` teal) reserved for:** the primary **Claim** CTA fill, the primary **Confirm-Arrival / advance** CTA fill, the **active bottom-nav tab** (icon + label + indicator), **details links** ("View details" / route link, teal underline), the `RouteMotif` **Transfer Badge** circle, the `LifecycleStepper` **completed-step** teal circle + connector, and the logo chip. NOT for arbitrary decoration, not on every interactive element.

**Status / lifecycle colours** are owned by the shared components (do NOT re-map):
- The Available claim card status badge is the masked-pool state — rows are `status='paid' AND driver_id IS NULL`. Per DS-02 the mockup's "Unclaimed" label maps to a **coral** badge (Unclaimed = coral) with the worded label "Unclaimed". (Implementation note: the current `PoolView` renders `StatusDot state="paid"`; the rebuild presents these unclaimed-pool rows as the DS-02 **Unclaimed = coral** badge to match the mockup and DUI-01 — a presentation choice over the same masked data, no data change.)
- My Trips trip cards use `StatusDot` for each row's real state: Claimed=teal, En route=amber, Completed=grey, Cancelled=hollow coral ring. Always coloured dot/badge **+ worded label** (WCAG 1.4.1).
- The trip-detail `LifecycleStepper` encodes completed (teal + check) / active (amber) / pending (grey outline ring) by **shape**, never colour alone; each step keeps its worded label.

**Elevation:** tonal, not heavy (DESIGN.md). Cards keep the existing 1px `#66676F`@~30% border (`border-grey/30`) + soft `shadow-sm` (the Phase 9 ambient `0 4px 12px rgba(47,72,88,0.08)` is the heavier permitted option for the detail hero card). The bottom-nav bar gets a 1px top border (`border-grey/20`) and may carry a subtle upward ambient shadow. Buttons are solid fills, no gradients.

**Radii** (from `@theme`): claim/trip cards + detail cards `rounded-md` (12px) — matching the existing driver cards; large hero/detail container may use `rounded-lg` (16px); CTAs `rounded-md`/`rounded-lg` (existing `Button`); status dots + Transfer Badge `rounded-full`; the "Unclaimed" pill badge `rounded-full`.

## Copywriting Contract

All user-facing strings go through the EN/BG dictionary parity gate (`platform/i18n/en.ts` + `bg.ts` — tsc fails on a missing key). **Every NEW key (bottom-nav labels, "Unclaimed" badge, My Trips title/empty, trip-card captions, passenger-info captions, Confirm-Arrival CTA) MUST be added to BOTH `en.ts` and `bg.ts`.** Reuse existing driver keys verbatim where they already exist (`claimTransferCta`, `claimLostToast`, `claimFailedToast`, `poolEmptyHeading`/`poolEmptyBody`, `myRunTitle`, `runEmptyHeading`/`runEmptyBody`, `completedTodayTitle`, `advanceTo*Cta`, `advanceFailedToast`, `airportLabel`, `zoneLabel`, `addressLabel`) — no claim/advance behaviour or copy semantics change.

| Element | Copy |
|---------|------|
| Primary CTA — Claim | **"Claim transfer"** (reuse existing `claimTransferCta`). Teal fill, 52px. Invokes the unchanged `claimAction(id)`. |
| Primary CTA — Confirm-Arrival (detail) | **"Confirm arrival"** (new key `driverConfirmArrivalCta` for the en_route→arrived edge; other edges reuse the existing `advanceTo*Cta` labels). Wired to the existing `advanceStatus(id)`. |
| Claim — already-claimed (lost) | **"Already claimed by another driver."** — reuse existing `claimLostToast`, **neutral** tone (NOT an error, D-03). The card is silently removed. (DUI-05: first-to-claim-wins reflected gracefully.) |
| Claim — failure | Reuse existing `claimFailedToast` ("Couldn't claim — please try again."), **coral error** tone; card stays. |
| Advance — failure | Reuse existing `advanceFailedToast`, coral error tone. |
| Unclaimed badge | **"Unclaimed"** (new key `driverUnclaimedBadge`) — coral DS-02 badge + worded label on every Available card. |
| Bottom-nav labels | **"Available"** (`navAvailable`), **"My Trips"** (`navMyTrips`), **"Profile"** (`navProfile`) — new keys, each paired with a line-icon (icon is the redundant non-text cue). |
| My Trips title | **"My Trips"** (reuse `myRunTitle` if its value matches; otherwise add `myTripsTitle`). Page heading (Display 28px). |
| Available empty state | Heading reuse `poolEmptyHeading` ("No transfers available"); body reuse `poolEmptyBody` ("New transfers will appear here when guests book and pay."). |
| My Trips empty state | Heading reuse `runEmptyHeading` ("No active trips"); body reuse `runEmptyBody` ("Claim a transfer from Available to start a trip."). |
| Trip-card captions | "Date", "Route", "Guests", "Duration" (or reuse arrival fmt) — new keys only where not already present (`driverTripDate` / `driverTripDuration` etc.). NO earnings, NO ratings caption (Decision 1). |
| Passenger-info captions (detail, post-claim) | Reuse existing detail `Fact` labels: guest name, guest phone, notes, airport, zone, address, arrival, flight, fare, passengers, luggage. These are legitimately visible post-claim (claiming-driver RLS). |
| Error state (generic) | "Something went wrong. Please try again." — shared inline error; pairs problem + retry. Field/validation copy unchanged. |
| Destructive confirmation | **None.** No destructive driver action exists on this surface — there is deliberately NO un-claim / give-back control (CLAIM-04). Cancelled is a **rendered status** (hollow coral ring + label), not a driver action. |

Stepper labels (from `LifecycleStepper` / `STEPPER_ORDER`, fixed): **Paid → Claimed → En route → Arrived → Picked up → Completed**. Cancelled short-circuits to the hollow-coral-ring terminal treatment. Do not hand-roll labels — the component derives them from `stateLabel()`.
