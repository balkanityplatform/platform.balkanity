<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin browser → admin RSC route | Every admin route re-gates `getCurrentRole() === "admin"` server-side BEFORE any read; UNCHANGED by this plan (the gate stays on each page, not the layout). |
| layout RSC → notifications feed | The shell seeds the bell via `readOwnNotifications()` — caller-auth own-rows RLS, the same read the pages already used; relocation, not widening. |
| layout RSC → auth session | Signed-in identity read via `auth.getUser()` (revalidated JWT), never `getSession()`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Elevation of Privilege | app/admin/layout.tsx | mitigate | Layout adds NO role gate and NO service-role client; each page keeps its `getCurrentRole()==='admin'` gate before its read (preservation). Grep gate confirms no `getCurrentRole`/`service.role` in the layout. |
| T-12-02 | Information Disclosure | NotificationBell seed | mitigate | Bell seeded once via `readOwnNotifications()` (own-rows RLS) — never service-role, never another user's rows. Grep gate confirms exactly one `readOwnNotifications` and zero service-role refs. |
| T-12-03 | Spoofing | signed-in identity | mitigate | Identity read from `auth.getUser()` (revalidates the JWT), never the cookie-trusting `getSession()`. |
| T-12-04 | Information Disclosure | top-bar client search | accept | Search input is presentational here; it will only ever filter rows the RLS read already returned (wiring lands in Plan 03). No new data path; no PII beyond the existing admin read. |
| T-12-SC | Tampering | npm/pip/cargo installs | mitigate | This plan installs NO packages (consumes only existing deps). No install task → no slopcheck checkpoint required. |
</threat_model>
